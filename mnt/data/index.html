Interactive Guide to ConfigMap Volumes

Learn how to decouple application configuration from container images in Kubernetes.

What You'll Accomplish

In this lab, you will deploy a simple Python web application. Instead of hard-coding its greeting message, you will inject the configuration using a `ConfigMap` mounted as a volume. This is a fundamental pattern for building flexible and maintainable applications on Kubernetes. Click the buttons in each step to proceed through the lab.

Step 1: Define and Create the Configuration

First, we define our application's configuration in a simple properties file. Then, we create a `ConfigMap` object within Kubernetes to hold this data. The `ConfigMap` stores our configuration, making it available to pods within the cluster.

GREETING=Hello from a Kubernetes ConfigMap! This is much better than hard-coding.

data:
  config.properties: |
    GREETING=Hello from a Kubernetes...

Step 2: Deploy the Application

Next, we define a `Deployment`. This tells Kubernetes to run our Python application container. The crucial part is the `volumes` section, where we mount our `app-config` ConfigMap into the container's filesystem at a specific path.

...
# Mount the volume into the container
volumeMounts:
- name: config-volume
  mountPath: /app/config.properties
  subPath: config.properties
...
# Define the volume using our ConfigMap
volumes:
- name: config-volume
  configMap:
    name: app-config
...

✓ Deployment Created & Pod Running

Pod: `greeting-app-pod-xxxx`
Container: `web-server`

Filesystem:

↳ /app/config.properties (from ConfigMap)

Step 3: Expose and Verify the Result

Finally, we create a `Service` to expose our application to network traffic. We can then send a request to our application and verify that it's serving the custom greeting message loaded from the `ConfigMap`.

> curl http://greeting-app-service

The application correctly read the configuration from the mounted volume. You have successfully decoupled the app's configuration from its container image.
